# GitHub Score Comment Design

## Goal

게임 오버 시 플레이어가 GitHub 계정으로 로그인한 뒤, 서버 없이 고정된 `Leaderboard` 이슈에 자신의 플레이 결과를 댓글로 기록할 수 있게 한다. 이 흐름은 GitHub Pages에 배포된 정적 프론트엔드에서 동작해야 하며, 게임 플레이 자체를 방해하지 않고 게임 오버 이후에만 선택적으로 실행된다.

## User Experience

게임 오버 오버레이에는 기존 `Restart` 버튼과 함께 `GitHub에 점수 기록` 보조 버튼이 노출된다. 사용자가 버튼을 누르면 현재 점수에 대한 기록 요청이 시작되고, 브라우저는 GitHub App 인증 화면으로 이동하거나 팝업을 연다. 인증이 완료되면 사용자는 다시 게임 페이지로 돌아오고, 앱은 자동으로 고정된 `Leaderboard` 이슈에 댓글을 작성한다.

기록 상태는 오버레이 안에서 짧고 명확하게 보여준다. 상태는 `로그인 필요`, `GitHub 로그인 중`, `점수 기록 중`, `기록 완료`, `기록 실패` 중 하나이며, 성공한 점수는 같은 세션에서 다시 제출하지 못하게 잠근다. 실패한 경우에만 다시 시도할 수 있다.

## Recommended Approach

권장 구현은 `GitHub App + PKCE + REST API 댓글 작성`이다. 브라우저는 GitHub App의 클라이언트 ID를 사용해 PKCE 인증을 시작하고, 리다이렉트로 돌아온 뒤 코드 교환을 통해 사용자 액세스 토큰을 얻는다. 토큰을 획득한 뒤 `POST /repos/{owner}/{repo}/issues/{issue_number}/comments` 호출로 고정 이슈에 댓글을 추가한다.

이 방식은 사용자 GitHub 계정으로 기록을 남길 수 있고, GitHub REST API의 CORS 지원 덕분에 정적 사이트에서도 호출 가능하다. 다만 클라이언트 앱은 공개 클라이언트이므로 토큰 저장을 최소화해야 하며, 본 설계에서는 토큰을 장기 보관하지 않고 현재 제출 흐름에만 사용한다. 구현 시 GitHub App 설정에는 GitHub Pages 배포 주소에 맞는 정확한 리다이렉트 URL과 사용자 권한 범위가 필요하다.

## Alternatives Considered

첫 번째 대안은 GitHub OAuth App을 사용하는 것이다. 동작 자체는 비슷하지만 새 구현 기준으로는 GitHub App 쪽이 권한 모델과 최신 문서 흐름에 더 잘 맞으므로 채택하지 않는다.

두 번째 대안은 GitHub 댓글 작성 페이지로 이동해 사용자가 직접 제출하도록 하는 것이다. 구현은 간단하지만 `utterances` 같은 자동 기록 경험이 깨지고, 사용자가 직접 제출 단계를 마쳐야 하므로 채택하지 않는다.

세 번째 대안은 매 게임마다 새 이슈를 생성하는 것이다. 자동화는 가능하지만 저장소 이슈 목록이 빠르게 오염되고 리더보드 성격의 기록 모음으로도 부적합하므로 채택하지 않는다.

## Data Model

댓글 본문에는 아래 데이터를 사람이 읽기 쉬운 텍스트 형식으로 기록한다.

- `Score`
- `Wave`
- `Lives`
- `Played at`
- `Build`

`Build`는 우선 `package.json`의 버전 문자열 또는 빌드 시 주입한 커밋 해시 중 사용 가능한 값을 넣는다. 사용자 식별은 별도 닉네임 입력 없이 GitHub 댓글 작성자의 계정으로 대체한다.

예시 형식:

```text
Score: 1240
Wave: 9
Lives: 0
Played at: 2026-04-19 23:10 KST
Build: 1.0.0
```

## Components

### Game Over Overlay

[src/main.js](D:/tower-defense/src/main.js)의 게임 오버 오버레이 렌더링 분기에 점수 기록 버튼과 상태 텍스트를 추가한다. 이 레이어는 사용자의 제출 진입점이며, 게임 상태를 바꾸지 않고 제출 전용 UI만 담당한다.

### GitHub Auth Module

브라우저 전용 인증 유틸리티를 별도 모듈로 분리한다. 이 모듈은 PKCE 코드 생성, `state` 생성, 인증 URL 구성, 리다이렉트 파라미터 해석, 토큰 획득, 토큰 폐기를 담당한다. 페이지 새로고침과 리다이렉트 복귀를 고려해 PKCE 검증용 값은 `sessionStorage`에 잠깐 저장한다.

### Score Submission Module

점수 제출 유틸리티는 현재 게임 상태에서 댓글 본문을 만들고, 인증된 사용자 토큰으로 GitHub 댓글 API를 호출한다. 전송 대상 저장소 소유자, 저장소 이름, 고정 이슈 번호, GitHub App 클라이언트 ID, 리다이렉트 URL은 코드 상수 또는 빌드 환경 변수로 관리한다.

## Data Flow

1. 플레이어가 게임 오버 화면에서 `GitHub에 점수 기록`을 누른다.
2. 앱은 현재 점수 스냅샷을 메모리에 고정하고, PKCE 인증을 시작한다.
3. GitHub 인증이 끝나면 브라우저는 설정된 리다이렉트 URL로 돌아온다.
4. 앱은 URL의 인증 결과를 해석하고 액세스 토큰을 확보한다.
5. 앱은 스냅샷된 점수로 댓글 본문을 생성한다.
6. 앱은 고정된 `Leaderboard` 이슈에 댓글을 작성한다.
7. 성공 시 UI를 `기록 완료`로 바꾸고 같은 점수의 중복 제출을 잠근다.
8. 실패 시 UI를 `기록 실패`로 바꾸고 재시도만 허용한다.

## Failure Handling

아래 실패 케이스를 명시적으로 처리한다.

- 사용자가 로그인 창을 닫거나 인증을 취소한 경우
- GitHub가 권한 승인을 거부한 경우
- 토큰 교환 또는 댓글 작성 API가 실패한 경우
- 이미 제출 중인데 버튼을 다시 누른 경우
- 게임이 재시작되어 기존 점수 스냅샷이 무효가 된 경우

각 경우 오버레이에 짧은 상태 메시지를 보여주고, 치명적 오류라도 게임을 깨뜨리지는 않는다. 인증 또는 API 실패는 모두 복구 가능한 제출 실패로 간주하고, 재시도는 같은 게임 오버 스냅샷에 대해서만 허용한다.

## Security and Trust Model

이 기능은 서버가 없으므로 점수 자체를 신뢰할 수 없다. 브라우저에서 계산한 점수를 브라우저가 그대로 GitHub에 쓰기 때문에 사용자가 개발자 도구로 값을 조작할 수 있다. 따라서 이 기능은 공식 경쟁 리더보드가 아니라, 플레이 기록을 GitHub에 남기는 소셜 기능으로 정의한다.

토큰은 장기 저장하지 않는다. 액세스 토큰은 현재 제출 흐름에서만 사용하고, 제출 성공 또는 실패 후 메모리와 `sessionStorage`에서 제거한다. 브라우저를 새로 열었을 때 로그인 상태가 유지되지 않아도 괜찮은 UX로 간주한다.

## Testing Strategy

게임 로직 테스트는 유지하고, 새 기능은 UI 및 인증 경계 위주로 검증한다.

- 게임 오버 상태에서만 점수 기록 버튼이 보이는지 확인
- 제출 성공 시 중복 클릭이 막히는지 확인
- 로그인 취소, API 실패, 댓글 성공 각각에 대해 상태 문구가 올바르게 변하는지 확인
- 댓글 본문이 현재 점수 스냅샷을 정확히 반영하는지 확인
- 리다이렉트 후 복귀 시 현재 제출 흐름이 정상 복원되는지 확인

자동 테스트는 순수 함수로 분리 가능한 부분부터 적용한다. 예를 들어 댓글 본문 생성기, 제출 상태 머신, URL 파라미터 파서 같은 로직은 단위 테스트 대상으로 분리한다.

## Implementation Notes

게임 로직 파일 [src/game/logic.js](D:/tower-defense/src/game/logic.js)는 점수 계산과 게임 오버 판정을 이미 제공하므로 가급적 수정하지 않는다. 기능 추가의 중심은 [src/main.js](D:/tower-defense/src/main.js)와 새 인증/제출 모듈이다.

GitHub Pages 배포 환경에 맞춰 리다이렉트 URL은 배포 절대 경로를 정확히 사용해야 한다. 한 저장소에 여러 환경을 두지 않는 한, 운영 배포 주소 하나만 우선 지원한다. 로컬 개발 중에는 인증 과정을 건너뛸 수 있는 개발용 가드 또는 명시적 목 모드를 둘 수 있다.

## Out of Scope

- 점수 무결성 검증
- 랭킹 정렬 및 상위 점수 집계 자동화
- 다중 저장소 또는 다중 리더보드 지원
- GitHub 외 소셜 로그인
- 플레이어 커스텀 닉네임 입력
