# Mobile Compact Control Bar Design

## Goal
Small-height mobile viewports에서도 조작 패드를 항상 유지하면서, 게임 보드 시야를 최대한 확보한다.

## Problem
현재 작은 모바일 viewport에서는 다음 세 요소가 동시에 높이를 차지한다.

- 게임 보드 프레임
- 오버레이 상태 카드
- 하단 고정 조작 패드

그 결과 세로 공간이 부족해질 때 보드 시야가 급격히 줄고, 하단 게임패드 일부가 잘리거나 겹쳐 보인다.

## Confirmed UX Priority
- 최우선: 보드를 최대한 많이 보여줄 것
- 제약: 조작 패드는 사라지면 안 됨
- 선택된 방향: 작은 높이에서는 `초압축 컨트롤 바`로 재구성

## Design Direction
작은 높이 viewport에서는 현재의 모바일 하단 패드를 “패널”이 아니라 “얇은 게임 인터페이스 바”로 취급한다.  
컨트롤은 항상 남아 있어야 하지만, 큰 카드형 구성을 유지하지 않고 보드 시야를 우선하는 밀도 높은 배치로 재편한다.

## Layout Strategy
- 기본 모바일 레이아웃은 유지한다.
- 특정 높이 이하에서는 `control-dock`의 전체 높이에 상한을 둔다.
- 하단 인터페이스는 3개 덩어리로 남긴다:
  - 짧은 상태 요약
  - 축소된 타워 선택 슬롯
  - 압축된 방향/액션 조작군
- 긴 설명 문장은 제거하거나 한 줄 요약으로 축약한다.
- 보드 프레임은 가능한 한 더 많은 세로 높이를 가져가도록 한다.

## Control Bar Behavior
- `dock-selection-summary`는 긴 설명형 문장 대신 현재 선택 상태만 짧게 표현한다.
  - 예: `Attack • 40G • Ready`
- 타워 선택 버튼은 큰 카드에서 작은 토큰 슬롯으로 축소한다.
- 방향 패드는 손가락 조작 가능성을 유지하되, 버튼 면적과 간격을 더 타이트하게 조정한다.
- 액션 버튼은 `Build`, `Upgrade`, `Pause`, `Restart`를 유지하되, 높이와 패딩을 줄인다.
- 아주 낮은 높이에서는 텍스트보다 아이콘/숫자 우선 구조로 전환할 수 있다.

## Responsive Trigger
- 현재의 너비 기준 모바일 전환 외에, 낮은 높이 viewport 전용 규칙을 추가한다.
- 이 규칙은 “모바일 너비 + 낮은 높이” 조합에서만 발동해야 한다.
- 데스크톱이나 충분히 높은 모바일 화면은 현재 UI 밀도를 유지한다.

## Visual Treatment
- 초압축 상태에서도 기존의 아케이드 보드 테마는 유지한다.
- 하지만 장식성보다 밀도와 가독성을 우선한다.
- 작은 높이 상태에서는 배경 장식, 여백, 패딩, 토큰 크기를 우선적으로 줄인다.

## Risks And Mitigations
- Risk: 지나친 압축으로 탭 정확도가 나빠질 수 있다.
  Mitigation: 버튼 수는 유지하되 불필요한 텍스트와 여백을 먼저 줄인다.
- Risk: 상태 요약이 너무 짧아져 이해가 어려울 수 있다.
  Mitigation: 비용, 선택 상태, 가능 여부 정도만 남기는 짧은 규칙형 문구를 사용한다.
- Risk: 오버레이와 컨트롤 바가 동시에 있을 때 다시 시야가 답답해질 수 있다.
  Mitigation: 오버레이 상태에서는 컨트롤 바를 가장 얇은 밀도로 유지한다.

## Validation
- 작은 높이 viewport에서 게임 보드가 현재보다 더 많이 보여야 한다.
- 하단 조작 패드는 사라지지 않아야 한다.
- 컨트롤 바는 일부가 잘리거나 화면 밖으로 밀리지 않아야 한다.
- 선택 요약은 짧지만 현재 상태를 이해할 수 있어야 한다.
